[1] 프로젝트 개요
--------------------------------------------------------------------------------

FabCanvas는 반도체 fab 개발단 엔지니어를 위한 데이터 분석 허브이다.
양산이 아닌 개발 단계(300mm, 월 1~3천장 규모)에서 공정 데이터를 수집,
분석, 시각화하여 엔지니어의 의사결정을 돕는다.

- 기술 스택: React+Vite (프론트엔드) + FastAPI+Polars (백엔드)
- 인프라: 웹서버(제한된 메모리, GPU 없음), 쿼리머신(~100GB RAM, GPU 없음)
- 네트워크: 외부 차단, 사내망만 사용
- AI 원칙: AI 없이 100% 독립 서비스 가능해야 함. AI는 편의 기능일 뿐.
- 상용화: 사내 도구로 시작 → 테라팹 등 외부 판매 가능성


[2] 핵심 도메인 용어 체계
--------------------------------------------------------------------------------

Step ID
  - 형식: 영문 2자 + 숫자 6자 (예: AA100010)
  - 제품별로 대부분 다르며, 같은 step_id면 같은 func_step을 의미

Func Step (기능 스텝)
  - 형식: 소수점 + 기능명 (예: "2.0 NW_PHOTO")
  - main step만 매칭. 매칭 테이블은 step_id + func_step 2열 구조
  - product 열 불필요 (step_id 자체가 제품별로 다름)

PPID (Process Plan ID)
  - 공정 레시피 코드

Knob
  - 실험 분류 체계

EQP_CHAMBER (장비_챔버)
  - 장비와 챔버 조합이 의미 있는 최소 단위
  - 챔버만 구분하는 것은 의미 없음
  - SPC에서 장비_챔버별 컬러링으로 한 화면에 비교


[3] 측정 및 분석 체계
--------------------------------------------------------------------------------

측정 종류
  - Inline 측정: 공정 중간 계측 (CD, OCD, thickness, overlay 등)
  - ET (Electrical Test): 웨이퍼 레벨 전기 특성
  - EDS (Electrical Die Sort): 다이 레벨 선별

SPC (통계적 공정 관리) - 개발 단계용
  - Trend % change
  - Historic High/Low
  - Spec Out
  - 박스플롯: median, P10, P90 차이 시각화
  - 장비_챔버별 컬러링 (15색 팔레트)

Wafer Map
  - 웨이퍼 위 다이별 패턴 시각화
  - 패턴 라벨링, 갤러리 기능

ML 분석
  - TabICL (CPU 전용), XGBoost, LightGBM
  - SHAP 기반 설명 가능성
  - GPU 없는 환경에서 동작해야 함


[4] 데이터 파이프라인 구조
--------------------------------------------------------------------------------

사내 환경:
  DataLake → 쿼리 VM(가상머신, ~100GB RAM)에서 데이터 추출
  → S3 드라이브에 parquet로 write
  → FabCanvas 웹서버가 주기적으로 S3 read

핵심 포인트:
  - API 연동이 아닌 S3 파일 기반 느슨한 결합
  - 쿼리 VM과 웹서버가 분리되어 있음
  - ETL 로직은 VM단에서 처리, FabCanvas는 정제된 데이터만 소비


[5] 어댑터 레이어 & 소프트랜딩 전략
--------------------------------------------------------------------------------

문제:
  - 사내 DataLake 스키마를 통째로 문서화할 수 없음
  - 테이블마다 컬럼명, 시간 컬럼 이름이 다름
  - 어떤 데이터가 어디 있는지는 현장 경험으로만 알 수 있음

해결 방향 - 온보딩 위자드:
  1. S3에 parquet 파일 등록
  2. FabCanvas가 자동으로 컬럼 스캔
  3. 타입 추론 (datetime, numeric, categorical)
  4. 역할 매핑 UI ("시간 컬럼은?", "wafer_id는?", "측정값은?")
  5. 매핑 저장 → 이후 자동 적용

기존 시스템 활용:
  - 매칭 테이블: step_id ↔ func_step 변환 (→ 전체 컬럼으로 확장)
  - Relation Hint: 3-tier 매칭 (exact/alias/substring) + 자동 캐스팅
  - Table Map: 테이블 간 관계 설정 UI

사내 도입 전략:
  - 한 번에 전체가 아닌 pain point 큰 기능부터 점진 도입
  - 첫 매핑은 개발자 본인이 설정, 이후 다른 엔지니어는 사용만
  - 환경별 차이는 .env 설정으로 흡수 (데모용 vs 사내용 동일 코드베이스)


[6] 공정 영역 태깅 (Process Area Tagging)
--------------------------------------------------------------------------------

목적:
  ML 영향성 분석 결과에서 "물리적으로 말이 되는 원인"과
  "통계적으로만 잡힌 허위 상관"을 구분하기 위함

2나노급 로직 공정 모듈 구분 (GAA Nanosheet 기준, 학계 공개 정보):
  - STI (Shallow Trench Isolation)
  - Well / VT Implant
  - PC (Gate Patterning) — Nanosheet release, dummy gate 등
  - Gate (HKMG) — High-K/Metal Gate 형성
  - Spacer
  - S/D Epi (Source/Drain Epitaxy)
  - MOL (Middle of Line) — Contact, via-to-gate 등
  - BEOL M1~Mn (Back End of Line) — Metal 배선

매칭 테이블 확장:
  - step_id + func_step + area(공정 영역) 3열
  - 데모: 학계 기준 일반 공정 흐름으로 area 매핑
  - 사내: 어댑터로 실제 step_id에 area 연결

ML 분석 활용:
  - SHAP importance를 area별로 그룹핑 → "BEOL 구간 기여도 높음" 해석
  - Y값의 area와 X feature의 area 비교 → 물리적 타당성 자동 판단


[7] 공정 인과 방향성 매트릭스 (Causal Direction Matrix)
--------------------------------------------------------------------------------

기본 원칙:
  - 앞 공정 step이 바로 뒤 step에 영향: 강함
  - 두세 단계 앞: 점점 감소
  - 뒤 공정이 앞 공정에 영향: 거의 없음

예외 (형상 전사 효과):
  - Gate 영역에서 poly 제거한 프로파일이 한참 뒤 metal fill 때 전사됨
  - 즉, 공정 거리가 멀어도 "물리적 형상 전사"가 있으면 영향 가능
  - 예: PC poly removal → BEOL metal fill (전사 효과로 인해 영향 존재)

영향 매트릭스 예시:

  [Source → Target]       영향도    이유
  ─────────────────────────────────────────────────
  PC → Gate               강함      직전 공정
  Gate → Spacer           강함      직전 공정
  Gate → MOL Contact      강함      형상 전사
  Gate → BEOL M1          중간      간접 전사
  PC → SD Epi             중간      2단계 앞
  PC → far BEOL           약함      거리 멀고 전사 경로 불분명
  BEOL → Gate             거의없음  역방향
  BEOL → PC               거의없음  역방향
  S/D Epi → MOL           강함      직전 공정
  MOL → BEOL M1           강함      직전 공정

ML 결과 필터링 적용:
  - Y값(예: PCCA lkg)의 공정 영역 확인 → PC 영역
  - X feature 중 "far BEOL" step이 원인으로 잡힘
  - 매트릭스 참조: BEOL → PC = "거의없음"
  - 결과: "통계적으로 잡혔지만 공정 인과상 직접 원인 가능성 낮음" 플래그
  - 반대로 "Gate step → PCCA lkg" = Gate → PC 인접 = 신뢰도 높음

신뢰도 등급:
  - 높음: 같은 영역 or 직전 영역 or 형상 전사 경로 존재
  - 중간: 2~3단계 앞, 간접 영향 가능
  - 낮음: 거리 멀고 전사 경로 없음
  - 의심: 역방향 (뒤 → 앞)


[8] DVC 파라미터 도메인 룰 테이블
--------------------------------------------------------------------------------

목적:
  디바이스 엔지니어의 상식적 해석 방향을 시스템에 선행 입력.
  SPC 트렌드나 ML 결과에 자동 해석 주석 달아줌.

파라미터     방향성              해석
──────────────────────────────────────────────────
Rc           lower_is_better     접촉저항. 낮을수록 소자 성능 향상
Rch          target_centered     채널저항. 목표값 대비 편차가 중요
ACint        lower_is_better     기생 인터커넥트 커패시턴스. 낮을수록 속도 개선
AChw         context_dependent   핫와이어 AC. 구조/목적에 따라 해석 다름
DC (Ioff)    lower_is_better     오프전류. 낮을수록 누설 적고 전력 효율 향상
DC (Ion)     higher_is_better    온전류. 높을수록 구동력 좋음
DC (Vth)     target_centered     문턱전압. 설계 타겟 대비 편차가 중요
lkg          lower_is_better     누설전류. 낮을수록 좋음

활용 시나리오:
  - SPC에서 Rc 상승 추세 감지 → "접촉저항 악화 방향, 성능 저하 우려" 자동 경고
  - ML에서 특정 step이 Rc에 영향 높다고 나옴
    → 인과 매트릭스 확인: 해당 step이 MOL Contact 영역 = 신뢰도 높음
    → DVC 룰 확인: Rc lower_is_better인데 해당 step이 Rc 증가 방향으로 영향
    → 자동 해석: "MOL step X가 접촉저항을 증가시키는 방향 — 성능 악화 원인 후보"


[9] 테이블 데이터 처리 규칙
--------------------------------------------------------------------------------

Dedup (중복 제거):
  - 모든 컬럼이 완전 동일한 행만 자동 제거
  - 부분 중복은 사용자 판단에 맡김

Relation Hint (테이블 간 연결):
  - 컬럼명이 달라도 매칭 테이블에 등록되어 있으면 같은 데이터
  - wafer_id 등 숫자/문자 혼재 가능 → join 시 자동 캐스팅(str↔int)
  - 3-tier 매칭: exact → alias → substring

데이터 타입:
  - 매칭에 등록된 컬럼들은 실제 같은 데이터, 이름만 다름
  - 타입 불일치(str vs int)는 자동 캐스팅으로 해결


[10] UX 원칙
--------------------------------------------------------------------------------

- 페이지 이동/탭 전환은 즉각 반응 (로딩 없이)
- 서버 쿼리, ML 처리 등 실제 대기만 로딩 표시
- keep-mounted 탭 (탭 전환 시 리렌더링 없음)
- stale-while-revalidate 캐싱
- 큰 테이블은 virtual scroll
- 불필요한 CDN 의존 제거 (pretendard 등 npm 직접 import)


[11] 아키텍처 원칙
--------------------------------------------------------------------------------

- AI 없이 100% 독립 동작 (수동 매핑, 수동 분석, 수동 분류 모두 가능)
- AI는 편의 기능 (자동 추천/자동 분류), 실패해도 수동 fallback
- 사내 오픈모델 수준에서도 동작하도록 프롬프트/파이프라인 단순하게
- 플러그인/어댑터 패턴으로 AI 구현체 교체 가능
- Docker 멀티스테이지 빌드, dev/prod 환경 분리
- GitHub 기반 코드 관리


[12] 향후 로드맵
--------------------------------------------------------------------------------

단기:
  - 쿼리 VM단 데모 구현 (데이터 추출 → S3 write)
  - ML Table VM단 데모 구현 (feature 생성 파이프라인)
  - 공정 영역 태깅 (step_id → area 매핑)
  - DVC 파라미터 룰 테이블

중기:
  - 공정 인과 방향성 매트릭스 (ML 결과 신뢰도 필터링)
  - 온보딩 위자드 (새 데이터 소스 자동 스캔 → 역할 매핑 UI)
  - 자동 리포팅 (ET 측정 → 메일 자동 전달)

목표:
  - 전체 데모 완성 후 테라팹 지원
  - "단순 데이터 도구"가 아닌 "반도체 공정 지식이 내장된 분석 플랫폼"
