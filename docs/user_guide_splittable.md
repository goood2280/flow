# 사용자 매뉴얼: SplitTable 페이지

> 문서 버전: 2026-05-03  
> 적용 범위: split 표 기반 plan/notes 운영 체계

## 1. 개요 — SplitTable이란?

SplitTable은 랏 단위의 plan/notes 작업을 효율적으로 편집하기 위한 표 기반 화면입니다.
하나의 큰 데이터를 “모듈/메트릭/필터” 단위로 분리해 보여주며, 공정 분석과 조치 기록을 동시에 다룹니다.

이 화면의 기본 철학은 다음과 같습니다.

- 고정된 wide 뷰를 넘는 유연성 제공
- lot 단위로 핵심 노트를 빠르게 등록
- 사용자별 권한 경로를 따르는 시트 편집
- 저장 단위 단위의 추적성 강화

## 2. 제품 선택

조회 대상은 제품별로 먼저 좁혀집니다.
초기 진입 시 `ML_TABLE_*` 패밀리 목록이 함께 노출되어 바로 선택할 수 있습니다.

### 2-1. ML_TABLE_* 선택

- ML_TABLE_RAW: 원시 입력/노트 확인
- ML_TABLE_SCORE: 점수 및 판독 테이블
- ML_TABLE_SUMMARY: 집계 결과
- ML_TABLE_ALERT: 경보/예외 중심
- ML_TABLE_IMAGE: 이미지/지표 연동

선택된 테이블은 다음 컬럼 집합, 소스 노출 순서, 정렬 기준에 바로 반영됩니다.

## 3. Lot 검색

Lot 검색은 root_lot_id 또는 fab_lot_id로 수행할 수 있습니다.

### 3-1. root_lot_id 검색

- 상위 추적단위 중심 조회
- 제품 연동이 강한 경우 사용 추천
- 하위 lot 집합을 일괄 확인하는데 유리

### 3-2. fab_lot_id 검색

- 작업 실행 단위 중심 조회
- 랏별 개별 상태를 추적할 때 유용
- 최근 수정일 정렬을 병행하면 작업 흐름 파악이 빠름

### 3-3. 검색 팁

1. 공백 제거 후 검색
2. 하이픈 포함 여부 일치
3. 대소문자 정책을 동일하게 유지
4. 검색 후 1회 이상 정렬 재적용

## 4. 컬럼 선택

컬럼은 기본 3가지 모드로 전환 가능합니다.

### 4-1. wide

모든 주요 지표를 한 번에 노출합니다.
대량 조회에 유리하고, 초기 파악용으로 적합합니다.

### 4-2. long

세로형으로 키-값을 확장해 보여줍니다.
디버깅이나 특정 항목 상세 점검에 적합합니다.

### 4-3. custom

사용자 지정 컬럼만 뽑아 저장 가능한 뷰입니다.
자주 쓰는 컬럼을 고정해 분석 반복성을 확보할 수 있습니다.

## 5. 소스 전환: KNOB / MASK / FAB / INLINE / VM / ET

SplitTable은 5개 핵심 소스의 데이터를 같은 인터페이스에서 비교할 수 있습니다.

- KNOB: knob 설정·변경 이력 성향 데이터
- MASK: 마스크/패턴 연계 지표
- FAB: 설비/공정 운영 관련 지표
- INLINE: 라인 내부 실시간 이벤트 성격 데이터
- VM: 장비 가상화·시뮬레이션 결과 기반 지표
- ET: 엔드 투 엔드 상태와 검사 결과

각 소스는 컬럼 가시성, 기본 정렬, 기본 단위가 조금씩 다르므로 소스 전환 시 저장 레이아웃 검토가 필요합니다.

## 6. 커스텀 세트

커스텀 세트를 통해 나만의 보기를 저장할 수 있습니다.

### 6-1. paste-sets

CSV/표현식 문자열을 붙여넣기해 임시 세트를 만들 수 있습니다.
컬럼 명이 공백이 있으면 정규화 규칙으로 치환됩니다.

### 6-2. customs

직접 만든 이름으로 저장한 사용자 뷰입니다.
운영자만 관리 권한이 있는 항목은 공유 시 주의하세요.

### 6-3. saved views

저장한 뷰는 팀 공유, 본인 기본값, 재열람 모드로 구분됩니다.

## 7. Notes / Save

### 7-1. 노트 작성

표의 행 단위 또는 셀 단위로 노트를 작성합니다.
노트는 나중에 감사 추적과 비교 분석에 쓰입니다.

- 간단한 상태 기록
- 변경 원인
- 조치 계획
- 후속 조치 예정일

### 7-2. plan 저장

plan은 의사결정 기록이며, 저장 시 다음이 함께 기록됩니다.

- 작성자
- 수정 시각
- 대상 lot / 모듈
- 전환 이전 값
- 저장 사유

## 8. override / debug

디버깅 모드에서는 특정 컬럼 계산식을 강제 override할 수 있습니다.
실제 생산 데이터 가공과 분리된 “검증 모드”로만 운영되어야 합니다.

- 예상과 실제 값 비교
- 소스 간 오차 추적
- 계산식 추적 로그 저장
- 버그 리포트 버튼으로 저장 내역 추적

override 적용 후에는 반드시 `save`와 `cancel` 흐름을 분리해서 운영하세요.

## 9. 25 wafer 풀 표시

25 wafer 풀 뷰는 wafer 단위 분석 시 유용합니다.
각 셀은 상태 플래그를 가지며, 다중 wafer를 일괄 확인할 수 있습니다.

### 9-1. 사용 시점

- 대량 lot의 품질 경향 파악
- lot-level 상태 편차 확인
- 공정별 오차 구간 비교

### 9-2. 주의점

- 전체 화면 확대/축소 시 포커스 유지
- 고해상도 화면 권장
- 실시간 갱신 지연에 따른 stale 값 표기 확인

## 10. FAQ 10개+

### Q1. wide와 long 중 어떤 모드가 빠른가요?

A1. 일반적으로 wide는 빠른 훑어보기, long은 상세 검증에 적합합니다.

### Q2. 커스텀 뷰가 저장되지 않아요.

A2. 동일 이름 충돌 및 권한 제한 가능성, 컬럼 ID 호환성 검증이 필요합니다.

### Q3. 25 wafer 풀에서 일부 칼라가 보이지 않습니다.

A3. 필터 상태(완료/진행/지연)와 뷰 레벨을 재설정해보세요.

### Q4. ML_TABLE_* 전환 후 정렬이 초기화됩니다.

A4. 데이터 소스별 메타가 다르므로 뷰 템플릿 재적용이 필요합니다.

### Q5. notes 입력 후 plan이 저장되지 않아요.

A5. 대상 lot가 잠김 상태인지, 권한 변경 중인지 확인 후 저장하세요.

### Q6. override 값이 즉시 반영되지 않아요.

A6. 디버그 버퍼 반영 타이밍이 있을 수 있어 동기화 버튼을 확인합니다.

### Q7. debug 모드에서 데이터가 어긋나요.

A7. 계산식의 단위 변환(%)와 샘플 집계 기준을 점검하세요.

### Q8. 대량 row에서 검색이 느려요.

A8. 검색 범위를 lot 1차 조건으로 좁히고 소스 단일 모드로 시도하세요.

### Q9. 저장한 뷰가 사라졌어요.

A9. 다른 브라우저 계정/기기 동기화를 먼저 확인하고, view scope(개인/팀)를 점검하세요.

### Q10. ET 데이터만 자주 누락돼요.

A10. ET 소스의 데이터 제공 구간과 캐시 갱신 주기를 점검하세요.

### Q11. 커스텀 셋팅을 다른 사용자와 공유하고 싶어요.

A11. 팀 공유 옵션이 있는지 뷰 메타에서 확인한 뒤 URL 기반 접근권을 제어하세요.

### Q12. root lot과 fab lot을 동시에 검색할 수 있나요?

A12. 검색은 단일 필드 중심이므로, 먼저 하나를 선택하고 전환 필터로 병행 조회하세요.

## 11. 추천 사용 흐름

1. 제품 선택
2. 소스 모드 지정 (KNOB/MASK/FAB/INLINE/VM/ET)
3. Lot 검색
4. wide 또는 custom 컬럼 세팅
5. 필요한 행에 노트 입력
6. plan 저장
7. 로그/변경 이력 확인

## 12. 운영 체크리스트

- 캐시 갱신 시간 기록
- override 사용 기록 별도 저장
- 팀 공유 뷰 권한 점검
- plan 저장 전 필수 항목(우선도/원인/요청사항) 입력

## 12-1. 고급 운영 노트

SplitTable를 장기적으로 안정화하려면 다음 패턴을 습관화하세요.

1. 조회 전 소스 하나만 먼저 고정한다.
2. wide 모드에서 샘플 20건 이상 확인 후 custom 저장.
3. lot 검색 시 root와 fab를 분리해 병렬 조회한다.
4. 노트는 원인과 조치만 담고, 추정은 별도 칼럼을 사용한다.
5. plan은 완료 예정일을 반드시 남겨 처리량 예측성을 높인다.
6. override는 디버그용 임시값으로만 사용하고 실 운영 값으로 복제하지 않는다.
7. ML_TABLE_* 전환 직후 데이터 타입 경고가 없는지 확인한다.
8. 컬럼 커스터마이즈는 팀 규칙과 충돌하지 않도록 공유 규칙을 둔다.
9. 저장 실패 시 에러 코드와 요청 payload를 함께 보관한다.
10. 동일 lot에 대해 반복 편집되는 사용자를 로그로 추적한다.
11. 새로 생성한 custom은 이름 접두사로 용도 구분한다.
12. 장기 분석은 table 뷰로 내보내기 후 외부 통계툴로 교차 확인한다.
13. 25 wafer 모드 진입 전 대상을 100개 이하로 축소한다.
14. debug 값은 변경 직후 스냅샷을 남겨 롤백 지점을 확보한다.

## 12-2. 배포 전 확인 체크

- custom view가 소스별로 동일하게 보이는지
- plan 저장권한과 사용자 계층이 일치하는지
- notes 저장 후 로그에 변경 전/후 값이 기록되는지
- override 모드에서 기본값이 노출되지 않는지
- 대형 lot 집합에서 메모리 사용률이 과도한지
- 결과 export 시 컬럼 순서가 일관된지
- S3/DB 기반 데이터의 최신 시점이 일치하는지
- 팀 공유 뷰가 의도한 범위만 노출되는지
