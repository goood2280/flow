# Product Philosophy

`flow`는 반도체 개발/pilot 단계에서 공정 데이터, 실험 plan, 이슈, 인폼, 회의, 액션을 한 화면권 안에서 이어 보기 위한 업무 플랫폼이다.

목표는 많은 기능을 모으는 것이 아니라, lot과 wafer를 중심으로 "무슨 일이 있었고, 지금 무엇을 해야 하는지"를 빠르게 판단하게 하는 것이다.

## 사용자 관점

주 사용자는 공정/개발/분석 담당자다. 이들은 예쁜 대시보드보다 아래를 원한다.

- 원천 데이터가 제대로 들어왔는지 확인
- 특정 lot/wafer의 상태를 빠르게 파악
- plan과 actual의 차이를 기록
- 이상 징후를 이슈와 인폼으로 연결
- 회의와 액션아이템을 운영 기록으로 남김
- 근거가 있는 분석 결과만 공유

## 제품 원칙

1. Lot/Wafer 중심
   - 모든 주요 화면은 가능하면 `product`, `root_lot_id`, `lot_id`, `wafer_id`로 이어져야 한다.

2. 운영 기록 우선
   - 분석 차트만 보여주지 않는다. 누가 어떤 판단을 했고, 어떤 후속 조치가 생겼는지 남겨야 한다.

3. 실데이터 soft landing
   - 환경마다 경로, 컬럼, 포맷이 다르다. 앱은 canonical 이름으로 내부를 유지하고, 차이는 adapter/profile에서 흡수한다.

4. AI 없이도 동작
   - AI는 보조 수단이다. 매핑, 편집, 분석, 공유는 수동 fallback이 있어야 한다.

5. 페이지는 도구여야 한다
   - landing page처럼 설명이 긴 화면보다, 반복 업무를 빠르게 수행하는 밀도 있는 화면이 맞다.

6. 변경 이력은 기능이다
   - plan, 회의록, tracker, inform, calendar는 모두 변경 주체와 시간을 남겨야 한다.

## 페이지 철학

- `FileBrowser`: 데이터가 있는지, 어떤 컬럼인지 확인하는 입구
- `SplitTable`: 실험 plan과 actual 차이를 lot/wafer 축으로 정리하는 작업대
- `Dashboard`: 운영 KPI와 SPC성 추세를 빠르게 보는 관제 화면
- `Tracker`: 막힌 일, 모니터링, 분석 액션을 닫을 때까지 추적하는 보드
- `Inform`: 제품/lot/wafer 단위 이슈를 담당자에게 전달하고 기록하는 로그
- `Meeting`: 논의, 결정, 액션아이템을 운영 데이터로 남기는 회의장
- `Calendar`: 결정/액션의 날짜와 상태를 모아 보는 일정판
- `Admin`: 사용자, 권한, 경로, 백업, 설정을 관리하는 운영 콘솔

## 좋은 기능의 기준

좋은 기능은 아래 질문에 답해야 한다.

- 사용자가 더 빨리 판단하는가?
- lot/wafer/product 맥락과 이어지는가?
- 실패했을 때 원인을 알 수 있는가?
- 권한과 공개 범위가 분명한가?
- 이후 회의, tracker, inform 기록과 연결되는가?

위 질문에 답하지 못하면 새 화면보다 기존 화면의 작은 개선이 낫다.
